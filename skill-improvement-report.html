源版本分析

两个版本各自做对了什么

在融合之前，先理解两个版本各自的核心贡献，避免在合并时把各自的优势磨掉。

可视化输出版（original_skill）

基于真实的 AI CLI 工具竞品分析项目（5 产品 × 20 触点）提炼，有充分的实践验证。

核心优势

  • HTML 报告视觉规格极具体：Snap Scroll 楼层、两层 Tab 导航、SVG 图标系统、CSS Token
  • 触点对比表格式规范清晰：列标题双层结构（产品名主 + 做法副）
  • TypeScript 开发规格格式（接口 + 状态机 + 事件响应），开发可直接落地
  • 协作模式参考表，明确 Human 和 AI 各自做什么

核心局限

  • 单一 UX 视角，PM/Tech/Strategy 场景无法覆盖
  • 5 阶段线性流程，用户无法中途调整或跳过
  • 没有 Human-in-the-Loop 检查点，过程不可控
  • 信息搜集方法论粗糙，没有图文采集、视频工作流、失败处理等细节

流程控制版（competitive-analysis-bufan）

4 阶段模块化设计，配套完整的 references 方法论文档，工程化程度更高。原版仅支持 Markdown/对话流输出，可视化能力在融合后引入。

核心优势

  • Human-in-the-Loop：10 个明确的等待确认检查点，过程完全可控
  • 图文采集协议：视觉验证、视频工作流、链接下钻、失败处理成体系
  • 长期跟踪支持：tracking/ + context/ 业务积累层，支持持续运营
  • Markdown 报告可独立交付，不强依赖 HTML 可视化阶段

核心局限

  • 原版无 HTML 可视化报告能力，仅以对话流/Markdown 形式输出
  • 分析视角隐式切换（AI 自主判断），用户无感知、无掌控
  • Stage 1 5 步骤中"目标"和"视角"重叠，"场景"和"维度"相邻且相关性高
  • Stage 3 分析步骤以 UX 旅程图为起点，对 PM/Tech/Strategy 视角不友好

融合策略

为什么这样融合

融合不是把两个文件合并，而是弄清楚「谁的问题意识更准确」，然后以此为判断标准取舍。

核心判断框架

过程结构

流程控制版更优

模块化的四阶段流水线解决了「用户不知道进行到哪了」的问题。10 个检查点让用户始终在掌控中。单一线性流程没有这个能力。

可视化框架

可视化输出版更优

单一 UX HTML 骨架 + P1 对比表是可视化能力的起点，融合版在此基础上扩展为 5 种视角骨架（S1-S5）和 10 种内容驱动的表达模式（P1-P10）。流程控制版原版无可视化能力。

信息搜集

流程控制版更优

图文必备要求、视觉验证协议、视频工作流、失败链接处理，这些细节决定了搜集结果的可追溯性。可视化输出版只有原则描述，没有落地操作规程。

输出规格

可视化输出版更优

对比表的列标题格式（产品名主 + 做法副）、TypeScript 开发规格格式，这些具体规格从真实项目里提炼，比流程控制版的概念性描述更落地。

协作模式

可视化输出版更优

Human 做「为什么」/ AI 做「怎么做」的协作模式参考表，帮助用户知道如何高效使用 Skill，是流程控制版原版所没有的。

视角机制

两版都不足

可视化输出版无视角概念；流程控制版有视角但隐式切换。用户不知道当前以什么角度在分析，结论的视角偏差难以被察觉。需要新设计：显性确认 + 贯穿全程。

一句话总结融合思路：以流程控制版为骨架（过程管控、方法论），以可视化输出版补血肉（HTML 规格、协作模式），以新设计修补两版的共同盲点（视角显性化）。
核心改动

改了什么，为什么这样改

改动 01 分析视角从隐式切换改为 Stage 1 显性确认
Stage 1.1 里，AI 在复述目标时同步推断并提出分析视角（UX/PM/UI/Tech/Strategy），等用户确认后贯穿全程。视角一旦确认，Stage 3 的分析步骤、Stage 4 的骨架类型都自动跟随，不再重新选型。同时去掉了"次视角"的概念——AI 内部可以引入其他视角的局部分析，但不向用户暴露这个决策。
早期流程中的"按需切换"对用户是黑盒。用户在看报告时不知道某个结论是从 PM 角度还是 UX 角度得出的，难以判断结论的适用边界。显性确认视角后，用户有掌控感，也可以在 Stage 1 就纠正方向，避免做完一整个分析才发现角度不对。去掉次视角是因为它会让 Stage 1 变复杂，而用户关心的是「报告能不能回答我的问题」，不是「用了哪些骨架组合」。
改动 02 Stage 1 从 5 步压缩为 3 步
原来 5 步：目标确认 → 竞品名单 → 场景 → 维度 → 规划。改为 3 步：
Step 1.1 目标 + 视角（合并） → Step 1.2 竞品 + 场景 + 维度（合并） → Step 1.3 生成规划。
两次合并的逻辑：目标和视角同时呈现更自然；场景和维度高度相关（场景定了，视角下的维度基本跟着来），分开问反而让用户觉得在重复表达同一件事。
5 步 Stage 1 会产生 5 次往返，用户还没开始分析就已经疲惫。每一步的信息本身没有问题，但是切分粒度过细会让每个检查点的决策量太小，用户反而不重视。3 步保持了必要的确认节奏，同时每一步的决策量都足够有意义。
改动 03 Stage 4 骨架选型改为自动推荐，不重新决策
Stage 4.1 不再展示"五种骨架类型让你选"的完整菜单。而是直接说：「基于你在 Stage 1 确认的 UX 视角，报告使用 S1 体验旅程型骨架，以下是具体页面方案……」用户确认页面方案，而不是重新选骨架类型。AI 在此处可以说明是否引入了其他视角的局部页面，但骨架主体已定。
用户在 Stage 1 已经做了视角决策，到 Stage 4 再面对一张"五种骨架"的选择表，会产生「我不是已经说过了吗」的疲惫感。骨架类型是对用户透明的实现细节，用户真正需要确认的是「这份报告的页面结构合不合理」，而不是「选哪个骨架」。
改动 04 加入"找独特数据点"的信息搜集原则
在 Stage 2 的核心原则里增加：「主动探寻独特数据点——搜集时特别留意竞品披露的具体数字（如用户数、效率提升百分比、功能使用率），这类数据在对比时极有价值。」
来自真实项目的实践经验。例如「Warp 数据显示 96% 的 diff 直接 Accept」这类数字，不是普通功能描述，而是能直接量化某个设计决策效果的证据。早期版本的搜集原则里没有对这类数据的专项提示，容易被忽略。
改动 05 明确图片优先级：真实截图 > Mockup > 链接，强调图文对照
在 Stage 2 核心原则里新增「图片优先，图文对照」条目，明确视觉证据的优先级：真实产品截图 > Mockup 复刻 > 来源链接。截图是一手证据，不依赖读者想象；Mockup 是截图不可得时的次选；纯链接最弱，需要读者自行点进去验证。同时要求最终报告做到图文对照——每条关键结论旁配对应图片，两者相互印证。在输出规范里也加入这条硬约束。
竞品分析报告里常见的问题是"文字说得有理，但读者不知道说的是哪个界面"。截图能让结论立刻可验证——读者扫一眼就知道你在说哪里、有没有夸张。纯 Mockup 是 AI 自己画的，可能和真实产品有偏差；纯链接要求读者配合打开验证，摩擦极高。这条原则把视觉证据的质量预期说清楚，避免用 Logo 图、媒体配图凑数。
改动 06 加入协作模式参考表 + 去掉 Terminal Mockup
在主 SKILL.md 末尾加入协作模式参考表，明确 Human 和 AI 在维度拆解、批量修正、内容填补、结构重构、合成升华五种模式下各做什么。
同时去掉 Terminal Mockup，理由是它只在 CLI / 终端产品分析时有用，放进通用 Skill 会增加噪音，需要时在领域特定扩展里单独加回来。
协作模式表帮助用户知道「怎么用这个 Skill 效率最高」，而不是靠试错摸索。Terminal Mockup 是在 AI CLI 研究项目背景下产生的领域特定产物，通用化后价值不大，保留反而会让用户在不适用的场景里尝试它。

改进效果

融合后的 Skill 能做什么

改进不只是把两个文件合并，而是形成了新的能力组合。以下从三个角度衡量效果。

覆盖范围更广

从 UX 单一视角扩展到 5 种视角（UX/PM/UI/Tech/Strategy），对应 5 种骨架类型和 10 种表达模式。一份 Skill 可以服务设计师、产品经理、技术负责人、战略分析师的不同需求，不再需要为每种场景单独写 Skill。

过程更可控

10 个 Human-in-the-Loop 检查点确保用户在关键决策节点都能介入和纠正。Stage 1 的视角显性确认让报告方向在最早期就对齐，避免做完一整份分析才发现视角不对，需要推倒重来。

输出质量更高

图文采集协议保证了素材的可追溯性和真实性；视觉核查防止用 Logo 图、媒体截图凑数；TypeScript 开发规格让报告可以直接驱动前端实现；协作模式表帮助用户更高效地使用 Skill。

各视角的覆盖能力

视角 原始 Skill 支持 融合版支持 典型使用场景

待修复：Stage 3 分析步骤仍以 UX 视角为主

当前 Stage 3 的核心步骤是：用户旅程地图 → 模式识别 → SWOT → 差距对比 → 洞察提炼。这个结构以 UX 视角为起点（旅程图是 UX 概念），PM 和 Tech 视角不需要从旅程图出发，强行套用会让分析偏离重心。

建议方向：为每个视角定义独立的 Stage 3 分析步骤序列，例如：

视角Stage 3 核心步骤
UX用户旅程地图 → 触点级对比 → 模式识别 → 差距矩阵 → 洞察
PM产品定位对比 → 功能覆盖矩阵 → 定价/商业模式分析 → 差距矩阵 → 决策洞察
UI视觉身份分析 → 色彩/字体/组件系统对比 → 动效/交互模式 → 设计语言洞察
Tech能力域映射 → API/集成分析 → 性能对比 → 开发者体验 → 技术洞察
Strategy市场定位图谱 → 竞争策略对比 → 用户声音分析 → 战略洞察

建议方向 A：领域特定扩展

competitive-analysis-bufan/references/ 里曾计划写 demo-conversations.md（三个完整 User Story），目前缺失。补充后可以让新用户快速理解 Skill 的交互节奏，也可以作为测试口令复用。对应三个场景：深度分析 / 快速摸底 / 继续历史分析。
