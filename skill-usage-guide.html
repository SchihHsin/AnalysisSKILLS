安装配置

三种使用方式配置

根据你的工作场景选择配置方式。方式 A / B 均基于 Claude Code，推荐优先使用；方式 C（Claude.ai 网页版）适合个人独立使用，无需开发环境。

方式 A · Claude Code CLI

1

创建命令文件

在项目根目录下创建以下结构，把 SKILL.md 的内容复制到命令文件里：

# 项目根目录下 .claude/ └── commands/ └── competitive-analysis.md ← SKILL.md 内容
2

放入 references 文件

将 references/ 目录放在项目内任意位置，Claude Code 可直接读取：

competitive-analysis/ └── references/ ├── research-methodology.md ├── analysis-frameworks.md ├── visualization-framework.md ├── verification-protocol.md └── assets/ ├── report-template.md └── tracking-template.md
3

用斜杠命令触发

/competitive-analysis ← 完整交互模式 /competitive-analysis fast Figma vs Sketch /competitive-analysis continue figma-d2c /competitive-analysis visualize notion-linear /competitive-analysis query Figma 和 Sketch 的定价

方式 B · IDE 插件

1

安装 Claude Code 插件

在 VS Code 或 JetBrains IDE 的扩展市场搜索并安装 Claude Code，登录 Anthropic 账号完成授权。

VS CodeExtensions 面板搜索 Claude Code
JetBrainsPlugins 市场搜索 Claude Code
2

放入 Skill 文件（与方式 A 相同）

IDE 插件与 Claude Code CLI 共用同一套命令文件，结构完全一致：

# 项目根目录 .claude/ └── commands/ └── competitive-analysis.md competitive-analysis/ └── references/ ← 可选，提升输出质量
3

在 IDE Chat 面板触发

打开侧边栏 Claude Chat 面板，输入斜杠命令，与方式 A 完全相同：

/competitive-analysis ← 完整交互模式 /competitive-analysis fast Figma vs Sketch /competitive-analysis query Figma 和 Sketch 的定价
IDE 插件优势：不切换窗口，分析过程中可直接引用当前编辑器里打开的文件（设计稿路径、配置文件等），生成的报告文件也直接出现在项目目录树里。

Stage 1 · 规划

目标确认 · 视角推断 · 竞品名单 · 分析规划

AI 在做什么

  • 检查 context/ 是否有业务背景，有则摘要告知
  • 检查 tracking/ 是否有历史分析可参考
  • Step 1.1：推断分析目标 + 视角，等你确认
  • Step 1.2：建议竞品名单 + 核心场景 + 分析维度
  • Step 1.3：生成完整规划，估计耗时，等你批准
  • 规划确认后保存到 tracking/{项目名}.md

你需要做什么

  • 说清楚这次分析要支持什么决策，越具体越好
  • Step 1.1：确认或纠正 AI 推断的视角
  • Step 1.2：增减竞品、调整场景和维度
  • Step 1.3：批准规划，说「可以开始了」
视角在这一步确认，之后不再重新选。确认视角直接决定 Stage 3 的分析维度和 Stage 4 的报告骨架类型——在 Step 1.1 如果觉得不对及时纠正，比做完再改省事得多。

Stage 2 · 信息搜集

多源搜集 · 截图采集 · 视频解析 · 失败处理

AI 在做什么

  • 按规划的竞品 × 维度逐一搜集资料
  • 优先官方文档，交叉核查关键数据
  • 每个竞品搜完后汇报进展（覆盖率 + 关键发现）
  • 遇到搜不到的信息，明确告知并给三个处理选项
  • 截图/文档/视频摘要保存到 materials/{项目名}/
  • 无法核查的数据标注 ⚠️ 待核查，绝不编造

你需要做什么

  • 等待，搜集期间可随时打断调整
  • 遇到信息缺口时做决策（跳过 / 换来源 / 自己提供）
  • 如有账号或截图，可直接发给 AI 补充
  • 搜集完成后确认进入分析
信息缺口很常见。竞品的部分功能可能未公开、需登录或已下线。AI 遇到这种情况会主动告知而非跳过或猜测，你来决定怎么处理。

来源优先级

P0官网、帮助文档、API 文档、更新日志
P1专业媒体测评、行业报告、白皮书
P2YouTube 测评视频、Reddit、论坛反馈
P3Dribbble / Behance 设计案例、产品截图
P4集成生态、合作伙伴、应用商店评论

视觉证据质量要求

每条关键结论必须配对应图片，做到图文对照。质量从高到低：

  • 真实产品截图（最优）— 一手证据，可直接核实
  • Mockup 复刻（次选）— 截图不可得时使用
  • 来源链接（最弱）— 需要读者自行点进去验证
拒绝用 Logo 图、媒体报道截图、文章配图凑数。

Stage 3 · 洞察分析

用户旅程 · 模式识别 · SWOT · 差距矩阵 · 洞察提炼

AI 在做什么

  • 绘制用户旅程地图（行为 / 触点 / 情绪 / 摩擦 / 机会，4 行结构）
  • 归纳竞品通用设计模式，识别独特设计语言
  • 为每个竞品做 SWOT 评估，附具体界面证据
  • 输出差距矩阵（功能差距 + 体验质量差距）
  • 提炼「如何做得不同且更好」的洞察与建议
  • 生成报告到 reports/{项目名}-{日期}-v1.md

你需要做什么

  • 收到旅程图后可确认或调整触点划分
  • 遇到设计分歧，AI 会先问你的立场再给分析
  • 报告生成后审阅摘要，决定是否要调整深度
  • 决定是否继续做 Stage 4 HTML 报告
每条洞察需满足三要素：证据（界面截图 / 数据）→ 分析（为什么这样设计）→ 建议（我们怎么做）。没有建议的洞察不算完成。

Stage 4 · 可视化 HTML 报告（可选）

骨架选型 · 多页生成 · Snap Scroll 导航 · Mockup 变体

AI 在做什么

  • 前置检查：确认 Stage 3 产物是否足够生成对应视角的骨架
  • 基于 Stage 1 确认的视角自动推荐骨架类型，不重新选型
  • 展示文件结构方案（index.html + 内容页 + highlights.html）
  • 逐文件生成，每完成一个汇报进度
  • 生成后自审 §8 质量清单，主动报告问题

你需要做什么

  • 确认页面结构方案（不需要重新选骨架，只确认具体页面合不合理）
  • 如需深色主题说「深色」，如需替换字体说明品牌字体名称
  • 最终审阅文件清单，打开 index.html 确认效果
  • 有问题直接说明，支持迭代修改
Stage 4 是独立可跳过的阶段。Stage 3 的 Markdown 报告已经是完整交付物，只在需要演示 / 对外分享 / 长期留存时才需要做 HTML。

功能全景

常见问题

视角推断不对怎么办？

在 Step 1.1 直接纠正：「不对，我想要 PM 视角，重点看功能和定价。」这是最好的纠正时机，后面改代价大。

某个竞品搜不到信息？

AI 会主动告知并给三个选项：跳过 / 换来源 / 你来提供。如果你有账号或截图，直接发给它补充效果最好。

报告太长 / 太短怎么调？

在 Stage 3 报告审阅环节说：「触点太多了，只保留最重要的 5 个」或「这个部分太浅，帮我深挖定价策略」。

能继续上次没做完的分析吗？

说「继续 [项目名] 的分析」，Skill 会读取 tracking/ 里的进度记录，从中断处恢复。项目名就是 tracking/ 里的文件名（不含 .md）。

HTML 报告做完想改怎么办？

直接说具体修改需求，支持：调整某个 mockup、修改亮点排序、增加对比维度（所有产品都会同步补充）、改配色、做成单文件等。

只想快速问一个问题？

说「快查：[你的问题]」，直接回答，不启动完整流程，不创建任何文件。适合临时查一个竞品的具体参数或价格。

协作模式

模式你做什么AI 做什么
维度拆解指出「这两个不是同一维度，原因是…」理解原则，找出所有同类问题一起拆
批量修正「所有 emoji 改成 SVG 线性图标」审计清单 → 批量替换 → 验证
内容填补「对应旅程地图补全缺失触点」自主 audit → 发现缺口 → 自动补充
结构重构「拆成多文件，用跨文件导航」设计架构 → 实现导航系统
合成升华「基于所有分析，输出设计方案」跨触点综合 → 提炼亮点 → 生成规格
给「为什么」比给「怎么做」更有价值。AI 理解了判断原则，会举一反三处理所有同类问题。
