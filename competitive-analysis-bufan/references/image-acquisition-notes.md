# 图片获取：遇到的问题与解决

## 遇到的问题与解决

### 1. 长时间无响应、像卡死

**现象**：点击「Dismiss」或截屏时，长时间没有返回，用户不知道是卡死还是在等。

**原因**：
- 页面加载或点击后 DOM 重绘，之前的元素 ref（如 `e6`）失效，自动化在查找元素时重试或挂起；
- 单次 `browser_wait_for` 设得过长（如 10s），期间没有任何中间反馈。

**解决**（已写入 [research-methodology.md](research-methodology.md)）：
- **短等待 + 再取 ref**：操作前先 `browser_wait_for` 2～3 秒，再 `browser_snapshot` 取**当前**页面元素 ref，再点击或截图，不依赖旧 ref；
- **避免单次过长等待**：用多次 2～3s 的短等待 + 快照检查，替代一次 10s 等待；
- **超时与跳过**：单页整体超时 30s，超时则跳过该链接并记入 `failed-links.md`，不阻塞后续。

---

### 2. 截到首屏而非正文关键图

**现象**：整页截图只截到导航/首屏，真正需要的产品图在正文里（如 `article-body > p[3] > img`），没有入镜。

**原因**：整页截图从视口顶部开始，长页面时正文内嵌图在下方，未进入截图范围或占比小；且截图慢、对滚动/分辨率敏感，不一定准。

**解决**（已写入 research-methodology §6）：
- **优先级**：**网页原有图片 > 区域截图 > 整页截图**；
- **先取页面内图**：下钻时解析正文区域（如 `article-body`、`.article-content`），提取 `img[src]` 直链，用 curl/fetch 下载保存，在 sources 注明「页面内图」；
- **整页截图仅作备用**：无合适内嵌图或需补全时再用；sources 中整页截图须备注「首屏/整页，非正文关键图」以免被误当主图使用。

---

### 3. 缺少系统的请求与失败处理

**现象**：下钻时没有去重、超时、重试、失败记录，容易重复抓、卡死、或丢失败信息。

**解决**（已写入 research-methodology「链接下钻：请求与失败处理策略」）：
- **加入任务 → 去重**：URL 已在 sources 或 links-to-drill「下钻后」标注则不再加入；
- **进入队列 → 限流**：每批 2～3 个链接；
- **发起请求 → 超时**：单页 30s，超时跳过并记录；
- **可重试错误 → 重试 + 退避**：最多 2 次，间隔 5s；
- **不可重试/多次失败 → 跳过 + 记录**：记入 `failed-links.md`（URL、原因、日期）；
- **大文件/超长视频**：可配置直接跳过（见可配置项表）。

---

### 4. 图文不一致：拿到的图与分析目标错位（🔴 强制规则）

**现象**：网络获取或截屏获得的图片，常与文本分析/搜索目标不一致——搜的是「产品 X 的 Y 功能 UI」，落地的素材却是该产品的 Logo、媒体报道里的人物照、文章插图、社交媒体讨论截图，甚至别的产品的界面。后续分析建立在错位的视觉证据上，结论失真。

**根因**：
- 仅凭链接、文件名、上下文文本就把图片纳入素材库，没有真正"看一眼"；
- 截屏时元素 ref 失效或截到了首屏/弹层，未确认截图内容；
- 搜索引擎/页面里的图本身就和正文话题错位（配图、广告位、相关推荐图）。

**强制解决规则**：

1. **目标必须是真实精准的产品 UI 界面**
   - 图片素材的目标对象**只能**是被分析产品的真实 UI 界面截图；
   - **拒绝**：媒体报道截图（含产品 Logo/创始人照片/新闻配图）、文章内嵌示意图、社交媒体讨论截图、PPT 截图、抽象示意图、广告物料；
   - 例外：当分析维度本身就是「市场声量 / 用户口碑」时，UGC 截图允许，但须明确标注用途。

2. **所有图片必须经多模态视觉验证后才纳入素材**
   - 不论是「页面内图直链下载」「区域截图」还是「整页截图」，**入库前必须由多模态模型实际加载该图并视觉判定**：
     > 这张图是不是 **[产品 X]** 的 **[分析维度 Y]** 的 **真实产品 UI 界面**？
   - 验证通过 → 写入 `sources.md`，文件名采用 `{product}-{feature}-{nn}.{ext}`；
   - 验证不通过 → **立即丢弃，重新获取**，不要"先存着以后再说"。

3. **网络图片链接同样必须视觉复核**
   - 抓到的 `img[src]` 直链，**不能仅凭 URL/alt/上下文文本认定内容**；
   - 必须 fetch 实际像素后，用多模态模型确认内容，再决定保存或丢弃。

4. **反复获取不到合格素材时，告知用户而非凑数**
   - 若同一目标连续 3 次抓取都未通过视觉验证：
     - 在 `failed-links.md` 记录「目标 / 已试链接 / 视觉验证失败原因」；
     - 在汇报中明确说：「[产品 X] 的 [功能 Y] **未获取到合格 UI 素材**，建议：① 由用户直接提供截图 ② 改用官网/Help Doc 中的官方插图 ③ 在报告中标注「视觉证据缺失」。」
   - **绝不**用 Logo 图、媒体配图、相似度低的界面图替代凑数。

**验证话术模板**（多模态确认时内部自问）：
```
图片：{filename / url}
预期目标：{product} 的 {feature} UI 界面
视觉判定：
  □ 是该产品的真实软件界面？（不是 Logo/配图/海报）
  □ 内容与目标功能匹配？（不是该产品的其他无关界面）
  □ 截图完整可读？（非半截/被弹层遮挡/像素糊到看不清）
结论：通过 / 不通过（原因：__________）
处理：入库 / 丢弃重抓
```

## 小结表

| 问题 | 原因 | 解决 |
|------|------|------|
| 长时间无响应 | ref 失效、单次长等待 | 短等待 + snapshot 再操作；超时 30s 跳过 |
| 截到首屏漏正文图 | 整页截图从顶部开始、慢且不准 | 页面内图优先，提取 img src 下载；整页截图仅备用 |
| 重复/卡死/失败无记录 | 无去重、无限流、无超时与重试策略 | 去重、限流、超时、重试退避、failed-links 记录 |
| 🔴 图文错位（拿到 Logo/媒体截图当 UI） | 仅凭链接/文本认定图片内容，未视觉确认 | 入库前用多模态视觉判定「是不是该产品的该功能真实 UI」；不通过丢弃重抓；3 次失败告知用户而非凑数 |
