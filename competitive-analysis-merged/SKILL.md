# AI 竞品分析

## 架构概览

```
四阶段流水线：
Stage 1 规划 → Stage 2 搜集 → Stage 3 分析（Markdown 报告）→ [可选] Stage 4 可视化报告（HTML）

数据层：
  tracking/     竞品跟踪表
  materials/    搜集素材（截图 / 文档 / 视频摘要）
  reports/      MD 报告 + 可视化报告

业务层：
  context/      产品背景 · 领域知识 · 持续更新
```

四个阶段相对独立：可单独更新跟踪表、单独补充背景、单独执行某阶段；可视化阶段可按需跳过。

### 分析视角（Stage 1 显性确认，贯穿全程）

| 视角 | 聚焦 | 默认骨架 |
|------|------|----------|
| UX | 任务流、信息架构、交互模式、可用性 | S1 体验旅程型 |
| PM | 功能范围、用户价值、商业模式、定价 | S2 功能全景型 |
| UI | 视觉系统、组件设计、动效、品牌表达 | S3 设计语言型 |
| Tech | 技术架构、性能、API、集成能力 | S4 能力架构型 |
| Strategy | 市场定位、竞争格局、差异化方向 | S5 竞争格局型 |

视角在 Stage 1.1 由用户确认，之后的分析维度、骨架类型、对比表行维度均跟随视角自动预设。AI 内部可根据内容引入其他视角的局部分析，无需再次询问用户。

## 意图路由

| $1 | 别名 | 动作 |
|---|---|---|
| 新建 | new | 以 $2.. 为初步目标进入 Stage 1 |
| 继续 | continue | 读 `tracking/$2.md` 与 `reports/$2-*.md`，判断是「恢复中断」还是「增量更新」（见下方说明）；$2 为空则列出候选 |
| 可视化 | visualize | 读 `reports/$2-*.md` 进 Stage 4，先做前置硬性检查，缺项回 Stage 3 |
| 快查 | query | 走快速查询，不写 tracking/、不生成 reports/ |
| 快速 | fast | 进快速模式（见下方说明） |
| 空 | — | 进 Stage 1 Step 1.1，完整 Interactive Mode |
| 其他 | — | 把 $ARGUMENTS 当自然语言，进 Stage 1，Step 1.1 复述时基于此展开 |

### 快速模式（fast）

适合已有明确目标、不需要逐步确认的用户。合并 Stage 1 的三步为一次性输出，搜集完成后直接进入分析，减少等待节点。

**触发方式**：
- 斜杠调用：`/competitive-analysis fast Figma vs Sketch`
- 自然语言：在需求里加「快速」「不用一步步确认」

**与标准模式的差异**：

| | 标准模式 | 快速模式 |
|---|---|---|
| Stage 1 | 3 步 × 3 次确认 | 1 次：AI 一次性给出目标+视角+范围+规划，等用户说「开始」 |
| Stage 2 开始 | 单独确认 | 用户说「开始」后直接进入 |
| Stage 3 开始 | 搜集完成后确认 | 直接进入，完成后给摘要 |
| Stage 4 | 骨架确认 + 设计系统确认 | 合并为一次确认 |

**注意**：快速模式下信息缺口决策和报告审阅检查点保留，这两处需要用户判断，不能省略。

### 继续模式（continue）— 恢复 vs. 增量更新

读取 `tracking/{project}.md` 后，判断属于哪种场景：

**场景 A：恢复中断**（上次分析未完成）
- 特征：tracking 里有未完成的 Stage 标记，无已有报告或报告是草稿
- 处理：汇报当前进度，询问从哪个 Stage 恢复

**场景 B：增量更新**（分析已完成，时间过去了，需要更新）
- 特征：已有完整报告（`reports/*.md`），用户重新触发 `continue`
- 处理：运行 diff 工作流

**增量 diff 工作流**（场景 B）：

```
发现上次分析完成于 {日期}，距今 {N} 天。

上次覆盖：{竞品列表} × {维度列表}

我来检查以下内容是否有变化：
□ 官网 / 产品更新日志（版本号 / 功能变更）
□ 定价页面
□ 关键页面截图对比（与 materials/ 里存档截图比较）

检查完成后，只更新有变化的部分，不重跑整个流程。
```

检查完成后汇报：

```
📊 变化检测结果

有变化：
· {竞品A} — {变化描述，附新旧对比}
· {竞品B} — {变化描述}

无变化：{竞品列表}

建议：{更新受影响的分析章节} / {重新生成报告 v{N+1}} / {仅更新 tracking}
怎么处理？
```

⏸ **等待用户决定更新范围**

## 交互原则（贯穿全流程）

这是一个设计师在用的工具。每次对话都要像一个有经验的设计研究伙伴：有主见、有逻辑、懂分寸、会汇报。

### 语气与风格
- **有观点**：提建议时附理由，不做没有判断的信息搬运工
- **不啰嗦**：一次只问一个核心问题，选项不超过 5 个
- **信息层级清晰**：用编号 / 加粗 / 分隔线区分主次
- **说人话**：用「我来帮你……」「我建议……」，避免机器腔

### Human-in-the-Loop 检查点

每个 `⏸ 等待确认` 标记处，**必须停下等用户回复，不得自动进入下一步**：

| 检查点 | 位置 | 触发 |
|--------|------|------|
| ⏸ 目标 + 视角确认 | Stage 1.1 结束 | 复述目标与推断视角，等用户确认或纠正 |
| ⏸ 范围确认 | Stage 1.2 结束 | 展示竞品 + 场景 + 维度，等用户增减 |
| ⏸ 规划确认 | Stage 1.3 结束 | 展示完整规划，等用户批准 |
| ⏸ 开始搜集确认 | Stage 2 开始前 | 说明预计耗时，等用户同意 |
| ⏸ 信息缺口决策 | 搜集遇到瓶颈 | 说明缺口，让用户决定如何处理 |
| ⏸ 开始分析确认 | Stage 3 开始前 | 汇报搜集结果摘要，等用户确认 |
| ⏸ 报告审阅 | 生成 Markdown 报告后 | 提供摘要，询问是否需要调整深度或方向 |
| ⏸ 可视化前置检查 | Stage 4 开始前 | 确认所需 Stage 3 产物就绪，否则先回 Stage 3 补齐 |
| ⏸ 骨架 + 页面方案确认 | Stage 4.1 | 展示骨架 + 页面方案，等用户确认 |
| ⏸ 可视化交付审阅 | Stage 4 完成后 | 提供文件清单 + 关键说明，询问是否需要调整 |

### 进度汇报格式

长时间操作中，每完成一个竞品的搜集，用固定格式汇报：

```
✅ [竞品名] 搜集完成
   · 官方资料：xxx（有 / 无）
   · 关键发现：{一句话}
   · 待核查：{有 / 无，有则列出}

⏳ 下一步：搜集 [竞品名]，预计 x 分钟
```

### 每次回答末尾的当前阶段信息

**每次回答的末尾**，带上当前阶段信息，让用户一眼看到进度：

```
---
**当前阶段**：Stage X — {阶段名}
**本步已完成**：{本条回复里做完的事，一句话}
**下一步**：{紧接着要做的事}
**等待**：{若在检查点则写「用户确认」；否则写「无」}
---
```

### 不确定时的处理
- 用户指令模糊 → **先复述理解，再问补充问题**，不要闷头猜
- 找不到信息 → **明确告知找不到什么、原因、替代方案**，不要跳过或编造
- 分析有分歧 → **提出两种解读，让用户选择**

## Stage 1 — 分析规划（Interactive Mode）

**核心规则：每个检查点必须等用户确认后才能继续，禁止连续推进多步。**

### 进入检查

1. 检查 `context/` 是否有相关业务背景 → 有则摘要告知用户（一句话）
2. 检查 `tracking/` 是否有已有跟踪列表 → 有则列出供参考

### Step 1.1 — 明确目标 + 视角

用自然语言提问，一次只问 **1 个核心问题**：

> 这次竞品分析最终要帮你做什么决策？
> 比如：功能优先级规划、某个流程的体验优化、市场全景摸底、还是找设计灵感？

收到回复后，**复述理解并推断分析视角**，等用户确认：

> 好的，我理解你的目标是：**{复述目标}**。
> 这次以 **{视角}** 为主——{一句话说为什么这个视角最合适}。
> 对吗？

**视角推断逻辑**（AI 内部判断）：
- 目标涉及"界面设计 / 交互流程 / 用户旅程 / 体验优化" → UX
- 目标涉及"功能对比 / 定价策略 / 商业模式 / 产品路线图" → PM
- 目标涉及"视觉设计 / 设计语言 / 品牌 / 组件风格" → UI
- 目标涉及"技术能力 / API / 性能 / 架构 / 集成" → Tech
- 目标涉及"市场定位 / 竞争格局 / 差异化战略" → Strategy

⏸ **等待用户确认目标与视角**

### Step 1.2 — 圈定范围

目标 + 视角确认后，在**同一条回复**里给出三项建议，请用户一次性确认或调整：

**竞品名单建议：**
- 直接竞品：A（理由，1 句话）/ B（理由）/ C（理由）
- 体验标杆：D（在 xx 方面领先，跨行业参考）

**核心场景建议：**
1. {场景一}（任务流描述，MECE）
2. {场景二}

**分析维度建议**（基于视角预设，最多 5 个，不贪多）：

| 视角 | 典型维度 |
|------|---------|
| UX | 步骤数 · 错误恢复路径 · 空态设计 · 帮助引导 · 操作反馈时机 |
| PM | 价格档位 · 免费配额 · 付费转化门槛 · 团队协作 · 企业级功能 |
| UI | 视觉密度 · 颜色语义 · 组件风格 · 动效哲学 · 信息层级策略 |
| Tech | 认证方式 · API 速率限制 · SDK 覆盖语言 · Webhook 支持 · 文档完整度 |
| Strategy | 目标用户段 · 增长策略 · 商业模式 · 定价杠杆 · 社区生态 |

> 竞品名单、场景、维度有需要调整的吗？

⏸ **等待用户确认范围**

### Step 1.3 — 生成分析规划

将所有确认结果整合，展示完整规划：

```
📋 分析规划已生成

目标：{目标}
视角：{视角}
竞品：{竞品列表，分类标注}
场景：{场景列表}
维度：{维度列表}
深度：{广域扫描 / 定向深挖 / 先广后深}
可信来源：{列出每个竞品推荐的官方文档 / 渠道}

准备好了可以开始搜集，预计耗时约 {时间估计}。
现在开始吗？还是有要修改的地方？
```

⏸ **等待用户批准规划**

批准后保存到 `tracking/{project-name}.md`，进入 Stage 2。

## Stage 2 — 信息搜集

详细方法论见 [references/research-methodology.md](references/research-methodology.md)。

**默认要求**：每个竞品、每个关键分析维度至少 1～2 张可追溯截图或界面证据；搜集过程中把高价值链接写入 `materials/{project}/links-to-drill.md`，再按优先级深度下钻。缺图须在素材清单与报告中标注并说明原因。

### 开始前告知

```
好的，规划已保存。现在开始搜集资料。

计划：搜集 {N} 个竞品 × {M} 个维度的资料
预计耗时：约 {X} 分钟
素材将保存到 materials/{project-name}/

我会在完成每个竞品后汇报进展，有任何问题可以随时打断我。
```

⏸ **等待用户确认（或直接收到「开始」指令后执行）**

### 核心原则

1. **多源交叉** — 官方一手 > 权威二手 > UGC；关键事实至少两个独立来源
2. **图片优先，图文对照** — 每个分析结论都应有对应的视觉证据，优先级从高到低：**真实产品截图 > Mockup 复刻 > 来源链接**。截图最优，因为它是一手证据，不依赖读者的想象；Mockup 是截图不可得时的次选；纯链接是最弱的证据，读者需要自己点进去核实。最终报告中每条关键结论旁须有对应图片，做到图文对照——先看图再读文字，或先读文字再看图，两者相互印证。
3. **视觉质量核查** — **所有入库图片必须经多模态视觉判定**是被分析产品的真实 UI 界面；拒绝 Logo 图、媒体报道截图、文章配图凑数；同一目标连续 3 次未获合格截图则在 `failed-links.md` 记录，告知用户补图方式
4. **主动探寻独特数据点** — 搜集时特别留意竞品披露的具体数字（如用户数、效率提升百分比、功能使用率），这类数据在对比时极有价值
5. **特性覆盖** — 用结构化清单防止遗漏重要特性
6. **素材留存** — 截图 / 文档 / 视频摘要保存到 `materials/{project-name}/`（含 `screenshots/`、`links-to-drill.md`、`sources.md`；失败链接记入 `failed-links.md`）
7. **事实标注** — 金额 / 时间 / 参数 / 版本号标注来源，不确定标「⚠️ 待核查」

### 搜索策略

| 优先级 | 来源类型 | 示例 |
|--------|----------|------|
| P0 | 官方资料 | 官网、帮助文档、API 文档、更新日志、官博 |
| P1 | 权威评测 | 专业媒体测评、行业报告、白皮书 |
| P2 | 用户声音 | YouTube 测评视频、Reddit 讨论、论坛反馈 |
| P3 | 设计资产 | Dribbble/Behance 设计案例、产品截图 |
| P4 | 生态情报 | 集成生态、合作伙伴、应用商店评论 |

### 遇到信息缺口

```
遇到一个情况需要你帮我决定：

❓ 无法获取的信息：[竞品X] 的 [具体内容] 找不到可信来源
   · 尝试了：官网、Google 搜索（中/英）、GitHub、App Store
   · 推测原因：{内测 / 未公开 / 需注册 / 已下线}

可以这样处理：
A. 跳过，在报告中标注「信息未能获取」
B. 换一个替代来源：[你的建议]
C. 这个信息你能直接提供吗？

怎么处理？
```

⏸ **等待用户决策**

### 截图视觉解析

搜集完成后、进入 Stage 3 前，对 `materials/{project}/screenshots/` 里的关键截图运行视觉分析——**直接用多模态视觉能力读图**，不依赖图片描述文字。

**每张截图提取以下维度**（按视角取舍，UI 视角全覆盖，其他视角选关键截图）：

| 维度 | 提取内容 |
|------|---------|
| 布局密度 | 信息层级数、留白比例、主视觉重心 |
| 色彩系统 | 主色 / 辅色 / 强调色，功能性颜色使用（错误 / 成功 / 警告） |
| 组件风格 | 按钮形态、表单样式、卡片 / 列表风格、导航模式 |
| 交互信号 | 状态提示、空态设计、加载态、操作反馈机制 |
| 独特决策 | 与行业常规明显不同的设计选择（无则留空） |

将结构化结果写入 `materials/{project}/visual-analysis.md`，格式：

```
## {竞品名} — {截图文件名}

场景：{对应哪个分析场景}
布局：{描述}
色彩：{描述}
组件：{描述}
交互信号：{描述}
独特决策：{描述 / 无}
---
```

**规则**：只分析真实产品 UI 截图；Logo 图、媒体报道截图跳过。不确定的标「? 待确认」，不推断。

### 搜集完成汇报

```
📦 搜集完成

覆盖率：{N}/{M} 项完整，{X} 项部分覆盖，{Y} 项未获取
保存素材：{总文件数} 个（screenshots + docs + video 摘要）
视觉解析：{已解析截图数} 张，写入 visual-analysis.md
⚠️ 待核查项：{列出或「无」}

整体来看，信息最充分的是 {竞品A}，{竞品B} 的 {维度} 资料有限。

接下来进入分析阶段，重点围绕 {最有价值的发现方向}。
可以开始了吗？
```

⏸ **等待用户确认进入 Stage 3**

## Stage 3 — 洞察分析

详细框架见 [references/analysis-frameworks.md](references/analysis-frameworks.md)。

### 分析过程中的互动节奏

分析不是闷头写报告，在关键分叉点主动与用户对话：
- 体验地图绘制完成后，可先分享「最值得关注的 1 个模式」，看用户是否认同再深入
- 遇到「竞品间有明显分歧」的设计决策 → 先问用户的直觉 / 立场，再给分析
- 分析中途发现「新问题」（不在规划内但很重要）→ 告知用户，由用户决定是否扩展范围

### 核心步骤

开始前先读取 `materials/{project}/visual-analysis.md`（如有），将视觉解析结论作为分析输入，避免在 Stage 3 重复描述已从截图提取的内容。

1. **用户旅程地图** — 按任务流绘制，标注触点 / 情绪 / 摩擦 / 机会（4 行结构，详见 analysis-frameworks.md §1）
2. **模式识别** — 归纳竞品通用设计模式 + 识别独特设计语言；UI 视角直接基于 visual-analysis.md 的「独特决策」字段展开
3. **SWOT 评估** — 每个竞品的 SWOT，含具体界面证据
4. **差距对比** — 差距矩阵（功能差距 + 体验差距），区分「行业标配缺失」和「体验质量差距」
5. **洞察提炼** — 提出「如何做得不同且更好」的具体方向与落地建议

### 洞察闭环

每条洞察必须满足三要素：
- **证据（What）**：界面截图 / 数据 / 用户反馈
- **分析（Why）**：为什么这样设计，受什么约束
- **建议（So what）**：我们怎么做，如何验证效果

### 洞察草稿确认（写报告前）

完成核心分析步骤后，**先不写完整报告**，提出 3 条最重要的洞察草稿，等用户确认方向再展开：

```
分析完成。在写完整报告前，先和你确认方向——

最重要的 3 个洞察方向：

1. {洞察标题}
   证据：{一句话，引用具体截图或数据}
   结论：{一句话}

2. {洞察标题}
   证据：{一句话}
   结论：{一句话}

3. {洞察标题}
   证据：{一句话}
   结论：{一句话}

这 3 个方向对吗？有要调整重心、去掉或补充的吗？
确认后我开始写完整报告。
```

⏸ **等待用户确认洞察方向**

用户确认（或调整）后再进入下面的「报告生成前确认」。

### 报告生成前确认

```
分析完成，发现了 {N} 个值得关注的洞察。

最重要的 3 个：
1. {洞察标题} — {一句话结论}
2. {洞察标题} — {一句话结论}
3. {洞察标题} — {一句话结论}

现在生成完整报告，预计 {X} 分钟。
报告会包含：完整分析 + 证据截图 + 优先级建议 + 可执行行动计划。
可以生成吗？还是有想调整的方向？
```

⏸ **等待用户确认**

### 报告输出

按 [references/assets/report-template.md](references/assets/report-template.md) 生成报告，保存到 `reports/{project-name}-{YYYY-MM-DD}-v{N}.md`。

报告生成后：

```
📄 报告已生成：reports/{filename}

{执行摘要，3 句话}

需要我对某个部分深挖，或者调整报告的重点方向吗？
```

⏸ **等待用户反馈**（开放式，不强制要求确认）

## Stage 4 — 可视化报告（可选）

详细规格见 [references/visualization-framework.md](references/visualization-framework.md)。

**这是一个独立可跳过的阶段。** Stage 3 已经产出完整 Markdown 报告；只在需要可视化交付物（评审 / 汇报 / 对外分享 / 长期资产）时才进入。

### 何时进入

- 用户显式请求：「做成可视化的」「生成 HTML 报告」「做成网页版」「做成可演示的版本」「做成 PPT / 演讲稿 / 分享版」
- Stage 3 完成后，用户对报告满意，主动询问交付形式

### 交付形态选择（进入 Stage 4 的第一步）

可视化有两种交付物，受众和场景不同，先让用户选：

```
报告可以做成两种形态——

A. 楼层滚动深度报告（HTML）
   多页楼层 + Snap Scroll + 两层导航，适合深读、评审、长期资产。

B. 横向翻页演讲 PPT（单文件 HTML）
   横向翻页 + 章节幕封 + 数据大字报，适合分享、路演、发布。
   由 guizang-ppt-skill 提供模板与风格（电子杂志风 / 瑞士国际主义风）。

C. 两者都要（先出 A 深度报告，再把核心结论浓缩成 B 演讲 PPT）

要哪种？
```

⏸ **等待用户选择交付形态**

- 选 **A** → 走下面的「前置硬性检查」→ Step 4.1～4.4（楼层报告流程，原有逻辑）
- 选 **B** → 走下面的「前置硬性检查」→ 跳转 [Step 4.PPT — 横向翻页演讲 PPT](#step-4ppt--横向翻页演讲-ppt)
- 选 **C** → 先完成 A 全流程，再执行 Step 4.PPT，PPT 内容取自已完成的楼层报告核心结论

### 前置硬性检查（不通过就不开始）

根据 Stage 1 确认的视角，对应骨架类型的前置数据必须就绪：

```
所有骨架通用（必须有）：
□ 综合洞察候选（跨分析单元的关键发现，每条含「为什么重要」）
□ 视觉证据（关键分析单元的截图或界面描述）

UX 视角 额外需要：
□ 用户旅程地图（阶段 + 行为 + 触点 + 情绪值 + 摩擦 / 机会）
□ 各触点的交互 / 流程分析数据

PM 视角 额外需要：
□ 产品定位 / 用户画像对比
□ 功能覆盖分析（按功能域分类整理）
□ 定价 / 商业模式分析

UI 视角 额外需要：
□ 各产品设计语言描述（色彩 / 字体 / 间距 / 组件风格）
□ 核心 UI 组件或交互模式的对比分析

Tech 视角 额外需要：
□ 能力评分数据（各维度可量化）
□ API / 集成分析（按技术域分类）

Strategy 视角 额外需要：
□ 可量化为坐标的定位分析（两轴定义清晰）
□ 市场 / 用户反馈数据
```

任何缺失项，告知用户缺什么、回 Stage 3 哪一步补齐，不要硬上。

⏸ **等待用户决定：补齐 / 用现有数据降级生成 / 取消**

### Step 4.1 — 骨架确认 + 页面方案

通过前置检查后，**基于 Stage 1 确认的视角自动推荐骨架**，不重新选型：

## Step 4.PPT — 横向翻页演讲 PPT

当用户在「交付形态选择」里选了 **B** 或 **C** 时执行。这一步**不自己手写 PPT 的 HTML/CSS**，而是委派给打包在仓库里的 guizang-ppt-skill，由它的模板与风格规范产出单文件横向翻页 PPT。

**Skill 位置**：[references/guizang-ppt-skill/SKILL.md](guizang-ppt-skill/SKILL.md)

### 执行流程

1. **加载 guizang skill**：读取 `references/guizang-ppt-skill/SKILL.md`，按其工作流执行。模板与资源都在该目录下：
   - 风格 A 模板：`references/guizang-ppt-skill/assets/template.html`
   - 风格 B 模板：`references/guizang-ppt-skill/assets/template-swiss.html`
   - 主题色 / 布局规范：`references/guizang-ppt-skill/references/`

2. **喂入分析产物作为 PPT 大纲**——不要让 guizang 从零澄清选题，直接把已完成的分析结论作为内容输入：
   - 封面：分析主题 + 竞品名单 + 视角
   - 章节：按 Stage 3 的核心洞察分章（每条洞察一节）
   - 数据大字报：搜集中拿到的关键数字（用户数、效率提升、占比等）
   - 图片网格：`materials/{project}/screenshots/` 里的关键界面截图
   - 收尾：差距结论 + 行动建议

3. **风格选择**——基于 Stage 1 视角给默认建议，仍让用户确认：
   - UI / UX / Strategy 视角 → 默认风格 A（电子杂志风，视觉感强）
   - PM / Tech 视角 → 默认风格 B（瑞士国际主义风，数据/信息驱动）

```
PPT 形态我建议用 guizang 的 {风格 A 电子杂志风 / 风格 B 瑞士风}，
因为这次是 {视角} 视角，{一句话理由}。

大纲（{N} 页）：
1. 封面 — {主题}
2. {洞察一}
3. {数据大字报 — 关键数字}
...
N. 行动建议

风格和大纲 OK 吗？
```

⏸ **等待用户确认风格 + 大纲**

4. **生成**：按 guizang 工作流产出单文件 HTML，保存到 `reports/{project}-{date}-slides.html`（与楼层报告目录 `reports/{project}-{date}-visual/` 区分开）。

5. **交付自审**：横向翻页是否正常（← →、滚轮、ESC 索引）、截图是否清晰、关键数字有无来源标注。

### 注意

- guizang 的来源标识（歸藏 / op7418）**只用于确认 skill 来源，不要写进生成的 PPT 页面、封面或配图**。
- PPT 是「最浓缩 5 分钟」，控制在合理页数，不要把楼层报告逐页搬过来。
- 事实核查规则照旧：PPT 里的数字同样要可追溯，不确定的不放。

## 协作模式参考

| 模式 | Human 做什么 | AI 做什么 |
|---|---|---|
| 维度拆解 | 指出「这两个不是同一维度，原因是……」 | 理解原则，找出所有同类问题一起拆分 |
| 批量修正 | 「所有 emoji 改成 SVG 线性图标」 | 审计清单 → 建图标库 → 脚本批量替换 → 验证 |
| 内容填补 | 「对应旅程地图补全缺失触点」 | 自主 audit 列表 → 发现缺口 → 自动补充 |
| 结构重构 | 「拆成多个文件，用跨文件导航」 | 设计文件架构 → 实现导航系统 → 维护一致性 |
| 合成升华 | 「基于所有分析，输出设计方案」 | 跨触点综合 → 提炼亮点 → 生成规格文档 |

**最重要的原则**：给「为什么」比给「怎么做」更有价值。AI 理解了判断原则，会举一反三处理所有同类问题。

## 核心约束

### 事实核查（贯穿全程）

详见 [references/verification-protocol.md](references/verification-protocol.md)。
- 关键数据标注来源链接，金额 / 参数 / 版本号需二次核查
- 无法验证的标注「⚠️ 待核查」，**绝不编造**
- 引用必须可追溯到原始页面

### 进度与体验
- 长时间操作前说明预估耗时
- 每完成一个阶段性节点，汇报当前进展和关键发现
- 遇到搜索瓶颈时主动告知并提出替代方案

### 输出规范
- 语言：中文
- **图文对照**：每条关键结论必须配对应视觉证据。优先级：真实截图 > Mockup > 链接。截图不可得时用 Mockup，Mockup 也无法覆盖时才退而求其次用链接并标注「缺图，见来源」
- 拒绝流水账式功能清单堆砌，要有分析深度
- 时效性：默认优先近 1 年信息
- **全局无装饰性 emoji**（技术文档内的 ASCII 字符除外）
