# Stage 3 — 分析框架与洞察提炼

## 分析执行流程

```
搜集素材（来自 Stage 2）
  │
  ├─ 1. 体验地图绘制（按场景/任务流）
  ├─ 2. 模式识别（归纳通用模式 + 独特设计）
  ├─ 3. SWOT 评估（每竞品，体验视角）
  ├─ 4. 差距对比（结构化矩阵）
  ├─ 5. 洞察提炼（机会点 + 可执行建议）
  └─ 6. 生成报告
```

## 1. 用户旅程地图（整个分析的骨架）

**这是整个分析的骨架。所有触点必须从这里生长出来。**

旅程图先回答「用户的完整使用路径长什么样、情绪在哪里掉下去」，再以此为锚点，让触点级竞品对比从旅程中长出来。常见错误是把竞品对比塞进旅程表——会让旅程退化为功能矩阵，既不利于看流程，也不利于看对比。**两者必须分开**：旅程图先画干净的「单产品/单场景」全景，触点级对比（见 §4 与 visualization-framework.md）才横向铺开多个竞品。

### 1.1 确定旅程阶段

通常 **4-6 个阶段**，覆盖完整使用周期。少于 4 个说明颗粒度太粗、看不出节奏；多于 6 个会让阅读者抓不到主线。

不同领域的阶段示例（仅作参考，须按当前场景重新切分）：

- **AI CLI / Agent**：启动 → 表达意图 → 等待执行 → 审查确认 → 迭代修正 → 收尾
- **电商**：发现 → 浏览 → 决策 → 购买 → 等待 → 收货 → 售后
- **SaaS**：注册 → 引导 → 核心任务 → 协作 → 数据导出 → 续订
- **移动 App**：首次启动 → 注册/登录 → 探索 → 核心使用 → 留存 → 流失/回归
- **设计/创作工具**：打开 → 创建 → 编辑 → 协作 → 导出 → 维护

### 1.2 旅程地图结构（固定 4 行，不要更多也不要更少）

| 行 | 内容 | 说明 |
|----|------|------|
| **用户行为** | 每阶段用户实际操作 | 具体动词，4-5 条；可加 SVG 线性图标，**不要堆 emoji** |
| **关键触点** | 该阶段涉及的 UX 决策点 | 这些触点定义了后续触点级对比的范围；可视化报告中应可点击跳转到对应触点页 |
| **情绪曲线** | SVG 折线图，标出最高/最低点 | **情绪最低谷 = 最大设计机会**，必须显式标注 |
| **摩擦 + 机会** | 当前问题与改进方向 | 红色 = 摩擦点（friction），绿色 = 机会点（opportunity） |

> **不要放「用户想法」行。** 保持旅程图干净，4 行已经信息量饱和。"用户想法"是访谈数据，不是从公开素材可推得的——除非有真实的用户访谈/客服记录/调研数据来源，否则编造的「用户想法」会污染整张地图。

### 1.3 触点识别原则

从旅程地图中提取触点，**每个触点必须同时满足三条**：

1. **对应一个具体交互节点**——是一个可见、可截图、可点击的 UI 行为单元，不是抽象概念
2. **可跨产品对比**——不同竞品在这个节点上做了**不同的设计选择**（如果所有竞品做法雷同，这个触点没有分析价值，剔除）
3. **有明确的好/坏判断维度**——能用至少一个标准说清楚「A 比 B 好/差/不同在哪」

### 1.4 维度拆分原则（关键，最容易出错的地方）

如果一个触点涉及多个不同维度，**必须拆分为多个独立触点**，否则对比表会塌陷成一锅乱炖、对比不出有效结论。

- ✗ 错误：**「输入框」**（位置 + 编辑能力 + 多模态输入 混在一起）
- ✓ 正确：拆为三个独立触点：
  - **输入框-位置/形态**——固定底部？浮层？嵌入对话流？
  - **输入框-编辑能力**——单行 vs 多行？快捷键支持？历史回溯？
  - **输入框-多模态**——是否支持粘贴图片、拖拽文件、语音输入？

**判断标准**：不同竞品在这个维度上是否做了**不同的设计决策**？如果答案是否，说明维度太细，可以合并；如果不同竞品的答案**无法用同一句话/同一个标尺描述**，说明需要拆分。

### 1.5 情绪曲线的绘制要求

情绪曲线不是装饰，是定位机会的工具。要求：

- **刻度**：每个阶段一个情绪值，建议 -2 到 +2 的整数刻度（沮丧/焦虑/中性/愉悦/惊喜）
- **依据**：基于用户反馈（评论、推文、客服记录、应用商店评分）+ 任务流摩擦点综合判断，**不能拍脑袋**——拍脑袋的情绪曲线会指向错误的机会点
- **必标注**：情绪最高点（标杆体验，可借鉴）+ 情绪最低点（机会点，必须深挖）
- **多产品叠加（可选）**：可叠加 2-3 条不同竞品的曲线，对比谁在哪个阶段更好；超过 3 条会让图变得难读

### 1.6 触点清单产出（衔接下一步对比与可视化）

旅程图绘制完成后，输出**结构化触点清单**，作为后续触点级对比与可视化报告的输入：

```yaml
journey:
  scenario: {场景名}
  stages:
    - id: stage-01
      name: {阶段名}
      user_actions: [{动作1}, {动作2}, ...]
      emotion: -1                          # -2 到 +2
      touchpoints:
        - id: tp-01-a
          name: {触点名}
          dimension: {该触点对应的单一维度}
          opportunity: high|medium|low
          friction: {一句话描述}
        - id: tp-01-b
          ...
```

**数量参考**（按研究深度选择）：

- 极简扫描：3-5 个触点
- 快速竞品调研：5-8 个关键触点
- 完整 UX 研究：15-25 个触点
- 大规模深度研究：25+ 个触点（建议拆分多个子项目）

> 触点清单是 Stage 3 与 Stage 4（可视化）之间的契约。**触点 id 必须稳定**，因为可视化报告中旅程图的触点链接、触点对比页的锚点、设计亮点页的引用都靠它串起来。

## 2. 模式识别

### 通用模式归纳

从多个竞品中提取共性设计模式：

```markdown
### 模式：{模式名称}
- **表现**：多个竞品都采用了xxx方式
- **适用场景**：当用户需要xxx时
- **竞品实例**：A 的做法 / B 的做法（附截图引用）
- **评价**：该模式有效的原因 / 局限性
```

### 独特设计识别

标注某竞品独有的设计语言或策略：

```markdown
### 竞品 A 独特之处
- **设计决策**：xxx
- **推测原因**：基于其目标用户/商业模式/技术能力
- **可借鉴度**：高/中/低（为什么）
```

## 3. SWOT 评估

从体验设计角度对每个竞品做 SWOT：

```markdown
## SWOT：竞品 A

| 维度 | 内容 | 证据 |
|------|------|------|
| **S 优势** | 交互流畅，xx操作仅需2步 | [截图1] |
| **W 劣势** | 信息层级混乱，新手找不到xx | [截图2] |
| **O 机会** | 该领域普遍缺少xx能力 | [来源链接] |
| **T 威胁** | 其xx交互已成用户习惯，难以挑战 | [用户反馈] |
```

SWOT 规则：
- 每个判断必须附具体证据（截图、数据、用户反馈）
- S/W 聚焦体验层面，不泛泛而谈"用户体验好/不好"
- O/T 要关联到我方产品可执行的动作

## 4. 差距对比

### 功能差距矩阵

```markdown
| 功能/能力 | 我们 | 竞品A | 竞品B | 差距说明 |
|-----------|------|-------|-------|----------|
| xx功能 | ✅ 基础 | ✅ 完善 | ❌ 无 | A领先，有xx细节 |
| yy能力 | ❌ 无 | ✅ 有 | ✅ 有 | 行业标配，我们缺失 |
| zz特性 | ✅ 有 | ❌ 无 | ❌ 无 | 我们的差异化优势 |
```

### 体验差距矩阵

```markdown
| 体验维度 | 我们 | 竞品A | 竞品B | 关键差异 |
|----------|------|-------|-------|----------|
| 首次上手耗时 | ~10min | ~3min | ~5min | A的引导最优 |
| 核心任务步骤 | 7步 | 4步 | 5步 | A精简了xx步骤 |
| 错误恢复 | 无引导 | 有回退 | 有撤销 | 我们缺失容错设计 |
```

### 竞品之间的差距

不仅看"我们 vs 竞品"，也分析竞品之间的差异，发现行业分歧点：
- 竞品 A 和 B 在 xx 方面采取了完全不同的策略，说明...
- 行业尚未形成统一范式的领域 = 创新空间

### 触点级深度对比（衔接可视化报告）

§4 上面三张矩阵是「整体俯瞰视图」。如果项目要进入 Stage 4 可视化报告，每个关键触点还须额外产出一张**触点级深度对比表**——这是可视化报告每个触点页面的直接数据源。

**表结构**（与上面的差距矩阵不同，这里**列是产品**，行是分析维度）：

```markdown
### 触点：{tp-id} {触点名称}

| 维度 | 竞品A — {做法一句话} | 竞品B — {做法一句话} | 竞品C — {做法一句话} |
|------|----------------------|----------------------|----------------------|
| 界面形态 | {做法描述 + 须配 mockup 或截图} | ... | ... |
| 核心机制 | {一句话说清做法} | ... | ... |
| 优势 | + {优势 1}<br>+ {优势 2} | ... | ... |
| 劣势 | − {劣势 1}<br>− {劣势 2} | ... | ... |
| 一句话总结 | {该产品在这个触点上的设计取舍} | ... | ... |

**核心张力**：{这个触点的设计矛盾，如「信息量 vs 界面清洁度」}
**推荐方向**：{我方该怎么做，含具体条件或数据}
```

**强制要求**：

1. **列标题双层结构**——主标题是产品名，副标题是「做法一句话」。不要只写产品名，副标题让读者扫一眼就能区分各家路线
2. **「界面形态」行必须有视觉**——mockup、UI 截面图、产品截图三选一，不能只有文字
3. **优势/劣势对称**——每家产品至少各 2 条优势 + 2 条劣势；只列优势不列劣势 = 没有真正分析
4. **「核心张力」必填**——每个触点都有设计矛盾，找不出来说明触点维度划分有问题（回到 §1.4 重新拆分）
5. **「无此功能」也是结论**——如果某产品没有这个触点的实现，照样在格子里写「无此功能」并在劣势行标注，不要留空

## 5. 洞察提炼

### 洞察闭环（每条必须完整）

```markdown
### 洞察 {序号}：{一句话标题}

**证据（What）**
在 {竞品} 的 {场景} 中观察到 {具体现象}。
[附截图/来源]

**分析（Why）**
这一设计选择背后的原因是 {用户需求/技术约束/商业考量}。
{与其他竞品对比，强化分析深度}

**建议（So what）**
- 可执行动作：{具体的设计/功能改进方向}
- 预期效果：{对哪个指标/体验维度有影响}
- 验证方式：{如何验证这个改进是否有效}
- 优先级建议：P0/P1/P2（理由）
```

### 机会点分类

将所有洞察归类：

| 类型 | 描述 | 行动建议 |
|------|------|----------|
| **快速跟进** | 行业标配但我们缺失 | 尽快补齐 |
| **体验优化** | 竞品做了但做得不好 | 做得更好 |
| **差异化创新** | 竞品都没做或做法分歧 | 探索新方案 |
| **防御巩固** | 我们的优势可能被追赶 | 加深护城河 |

## 6. 报告生成

按 `assets/report-template.md` 输出完整 Markdown 报告。

生成前检查：
- [ ] 每条洞察都有证据支撑
- [ ] 关键数据已标注来源
- [ ] 截图/引用可追溯
- [ ] 建议可执行且绑定了验证方式
- [ ] 无待核查项遗留（或已明确标注）
- [ ] 旅程图、触点清单、触点级对比表已结构化产出（如计划进入 Stage 4 可视化）

保存到 `reports/{project-name}-{date}.md`，并更新 `tracking/` 中的项目状态。

## 7. 衔接可视化报告（可选）

如果项目需要可视化交付物（如设计评审、跨团队对齐、对外分享），完成 Markdown 报告后可进入 **Stage 4 — 可视化报告**，详见 [visualization-framework.md](visualization-framework.md)。

**进入 Stage 4 的前置条件**：
- §1 用户旅程地图（含 4 行结构 + 触点 yaml 清单）已产出
- §4 触点级深度对比表已为每个关键触点完成
- §5 洞察已落地、可执行（用作 Stage 4 的「设计亮点」素材）

如果上面三项缺其一，先回 Stage 3 补齐，**不要在数据不全时硬上可视化**——会得到一个空壳子报告。
